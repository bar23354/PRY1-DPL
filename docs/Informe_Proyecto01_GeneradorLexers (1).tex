\section{Introducción}

El presente informe documenta el diseño y la implementación de una plataforma integrada para la generación, validación y visualización de analizadores léxicos a partir de especificaciones YALex. El sistema combina un motor de compilación léxica en Python, una API de servicios para integración y una interfaz web unificada orientada a la exploración visual del proceso completo de análisis léxico.

Un analizador léxico es la primera fase de un compilador o intérprete. Su responsabilidad consiste en transformar una secuencia de caracteres en una secuencia de tokens con estructura y significado sintáctico. Automatizar la construcción de analizadores léxicos a partir de reglas y expresiones regulares es un problema clásico que integra teoría de lenguajes formales, autómatas finitos, construcción directa de AFD y generación de código.

La plataforma desarrollada cubre el ciclo completo del problema: especificar reglas léxicas en formato \texttt{.yal}, parsear y normalizar dichas especificaciones, construir AFDs directos minimizados, generar un lexer Python autosuficiente, analizar entradas reales, visualizar autómatas y ejecutar lotes de casos de prueba desde una sola interfaz.

\section{Objetivos}

\subsection{Objetivo General}

Diseñar e implementar una plataforma integrada para la generación de analizadores léxicos basada en YALex, con backend en Python, API de servicios y frontend web moderno, validada con casos de complejidad baja, media y alta.

\subsection{Objetivos Específicos}

\begin{itemize}[leftmargin=2em]
  \item Implementar la construcción directa de AFD a partir de expresiones regulares mediante el método del árbol sintáctico aumentado.
  \item Implementar la minimización de AFD mediante refinamiento de particiones.
  \item Diseñar un parser para especificaciones YALex que soporte definiciones \texttt{let}, reglas, argumentos y bloques auxiliares.
  \item Generar archivos Python autosuficientes que funcionen como analizadores léxicos independientes del generador.
  \item Exponer las capacidades del motor léxico mediante una API tipada y consistente.
  \item Construir una interfaz web amigable y estética para compilar especificaciones, analizar entradas, visualizar autómatas y ejecutar casos de prueba.
  \item Validar el correcto funcionamiento del sistema con casos representativos de complejidad baja, media y alta.
\end{itemize}

\section{Alcance del Sistema}

La plataforma se organiza en cuatro módulos funcionales principales:

\begin{itemize}[leftmargin=2em]
  \item \textbf{Dashboard}: resume el estado general del sistema y sus métricas operativas.
  \item \textbf{Lexical Analysis}: permite seleccionar una complejidad, cargar o editar una entrada y obtener tokens, errores léxicos y estadísticas de análisis.
  \item \textbf{Lexical Generator}: permite cargar o editar una especificación YALex, compilarla, inspeccionar una regla activa, visualizar su autómata y descargar el lexer generado.
  \item \textbf{Test Case Manager}: permite listar casos de prueba, ejecutar casos individuales o lotes completos y mostrar resultados agregados de validación.
\end{itemize}

El sistema se concibe como una herramienta integral de experimentación, verificación y documentación del proceso de análisis léxico.

\section{Marco Teórico}

\subsection{Expresiones Regulares}

Las expresiones regulares constituyen un formalismo para describir conjuntos de cadenas, es decir, lenguajes regulares. En el contexto de la generación de analizadores léxicos, cada token se define por medio de una expresión regular que describe los lexemas aceptados por la regla correspondiente.

Los operadores fundamentales considerados en el sistema son:

\begin{itemize}[leftmargin=2em]
  \item \textbf{Unión} (\texttt{|}): representa alternancia entre expresiones.
  \item \textbf{Concatenación}: representa la yuxtaposición de subexpresiones.
  \item \textbf{Cerradura de Kleene} (\texttt{*}): cero o más repeticiones.
  \item \textbf{Cerradura positiva} (\texttt{+}): una o más repeticiones.
  \item \textbf{Opcionalidad} (\texttt{?}): cero o una ocurrencia.
\end{itemize}

En YALex se soportan, además, conjuntos de caracteres, rangos, conjuntos negados, literales simples y dobles, diferencia de conjuntos (\texttt{\#}), el comodín \texttt{\_} y el símbolo especial \texttt{eof}. La precedencia adoptada es, de mayor a menor: diferencia, operadores unarios, concatenación y unión.

\subsection{Autómatas Finitos Deterministas}

Un AFD se define formalmente como una 5-tupla $M = (Q, \Sigma, \delta, q_0, F)$, donde $Q$ es un conjunto finito de estados, $\Sigma$ es el alfabeto, $\delta$ es la función de transición, $q_0$ es el estado inicial y $F$ es el conjunto de estados de aceptación.

La relevancia de los AFD en este proyecto radica en que permiten ejecutar el reconocimiento léxico de forma determinista, eficiente y completamente independiente de librerías externas de expresiones regulares.

\subsection{Construcción Directa de AFD}

El método directo construye el AFD a partir del árbol sintáctico de la expresión regular aumentada $r\#$, sin pasar por un AFN intermedio. Para cada nodo del árbol se calculan cuatro funciones:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{anulable}($n$): indica si el subárbol puede generar la cadena vacía.
  \item \textbf{primeraPos}($n$): conjunto de posiciones que pueden aparecer primero.
  \item \textbf{últimaPos}($n$): conjunto de posiciones que pueden aparecer al final.
  \item \textbf{siguientePos}($p$): posiciones que pueden seguir inmediatamente a una posición dada.
\end{enumerate}

Cada estado del AFD se modela como un conjunto de posiciones del árbol. El estado inicial corresponde a \texttt{primeraPos} de la raíz, mientras que las transiciones se construyen a partir de la relación \texttt{siguientePos}.

\subsection{Minimización de AFD}

Una vez construido el AFD, se aplica un algoritmo de minimización por particiones. Dos estados son equivalentes si, para todo símbolo del alfabeto, transitan hacia estados pertenecientes a las mismas clases de equivalencia. El refinamiento iterativo de las particiones produce un AFD mínimo equivalente, reduciendo complejidad estructural y mejorando la claridad de la visualización resultante.

\subsection{Especificaciones YALex}

YALex (\emph{Yet Another Lex}) define un lenguaje de especificación para analizadores léxicos con la siguiente estructura general:

\begin{lstlisting}[style=yalexstyle, title={\small Estructura general de una especificación YALex}]
{ header }
let ident = regexp
...
rule entrypoint [arg1, ..., argn] =
    regexp { action }
  | regexp { action }
  | ...
{ trailer }
\end{lstlisting}

Las definiciones \texttt{let} permiten factorizar expresiones, mientras que la sección \texttt{rule} define patrones y acciones. La semántica de matching utilizada por el sistema es \emph{longest match}, con desempate por orden de declaración.

\subsection{Descripción General}

El frontend web presenta una experiencia integrada que centraliza las operaciones principales del sistema. La API expone contratos estables para los módulos visuales, mientras que el backend conserva la lógica formal del generador y del motor de escaneo. Esta separación permite mantener la exactitud del núcleo léxico en Python y, al mismo tiempo, ofrecer una interfaz moderna para exploración y validación.

\subsection{Parser y Normalizador YALex}

El parser YALex procesa comentarios, bloques \texttt{header} y \texttt{trailer}, definiciones \texttt{let}, entrypoints con argumentos y reglas separadas por \texttt{|}. A continuación, el normalizador convierte la sintaxis de alto nivel a una representación interna uniforme, respetando la precedencia de operadores y resolviendo referencias a definiciones previas.

\subsection{Construcción y Minimización de AFD}

Cada regla léxica se traduce a un árbol sintáctico aumentado, a partir del cual se obtiene un AFD directo. El autómata resultante se minimiza antes de ser serializado para el runtime o para la visualización del frontend.

\subsection{Runtime y Generación de Código}

El runtime implementa la política de \emph{longest match}: para cada posición de la entrada, evalúa todas las reglas disponibles, selecciona el prefijo más largo y, en caso de empate, prioriza la regla de menor índice. Las acciones asociadas se ejecutan de acuerdo con la especificación.

El generador de código produce un archivo Python que contiene:

\begin{itemize}[leftmargin=2em]
  \item los AFDs serializados como estructuras nativas;
  \item las clases y funciones necesarias para el escaneo;
  \item el entrypoint configurado por la especificación;
  \item las secciones auxiliares copiadas desde el archivo fuente.
\end{itemize}

\section{Diseño del Frontend}

El frontend se implementa dentro de la carpeta \texttt{frontend/} como una aplicación de \textbf{Next.js}, \textbf{React} y \textbf{TypeScript}. Su responsabilidad consiste en concentrar la visualización y operación de los módulos del sistema, mientras que el procesamiento formal, la compilación YALex y la ejecución del lexer se mantienen en el backend Python.

La base del proyecto frontend se define a partir de los siguientes archivos:

\begin{itemize}[leftmargin=2em]
  \item \texttt{frontend/package.json}: administra dependencias y scripts de desarrollo, pruebas y construcción.
  \item \texttt{frontend/next.config.ts}: configura Next.js y centraliza la reescritura de rutas relativas \texttt{/api/*} hacia el backend.
  \item \texttt{frontend/src/app/layout.tsx}: declara el layout raíz compartido por toda la aplicación.
  \item \texttt{frontend/src/app/page.tsx}: monta la vista principal y delega la interfaz al shell de navegación.
  \item \texttt{frontend/src/app/globals.css}: contiene estilos globales, definiciones visuales y reglas compartidas por todos los módulos.
\end{itemize}

La estructura funcional del frontend se organiza alrededor de un shell principal. Esta capa se implementa en:

\begin{itemize}[leftmargin=2em]
  \item \texttt{frontend/src/features/shell/app-shell.tsx}: compone la interfaz general del sistema.
  \item \texttt{frontend/src/features/shell/module-card.tsx}: representa los accesos a cada módulo desde la vista principal.
  \item \texttt{frontend/src/features/shell/module-panel.tsx}: decide qué módulo se renderiza en el panel activo.
  \item \texttt{frontend/src/hooks/use-active-view.ts}: mantiene el estado de la vista seleccionada.
  \item \texttt{frontend/src/lib/modules.ts}: define la metadata de los módulos visibles dentro de la interfaz.
\end{itemize}

La elección de esta estructura responde a una consideración técnica concreta: la interfaz debía funcionar como una sola aplicación integrada. Por ello, el cambio entre \emph{Dashboard}, \emph{Lexical Analysis}, \emph{Lexical Generator} y \emph{Test Case Manager} se resuelve dentro del mismo shell, sin delegarlo a documentos incrustados o a vistas aisladas.

Cada herramienta se implementa como una feature independiente dentro de \texttt{frontend/src/features/}:

\begin{itemize}[leftmargin=2em]
  \item \texttt{analysis/}: contiene \texttt{analysis-tool.tsx} y \texttt{use-analysis-tool.ts}, encargados de la entrada de texto, carga de archivos, ejecución del análisis y presentación de tokens y errores.
  \item \texttt{generator/}: contiene \texttt{generator-tool.tsx}, \texttt{use-generator-tool.ts}, \texttt{generator-graph.tsx} y \texttt{generator-templates.ts}; estos archivos resuelven la edición de especificaciones YALex, la compilación, la selección de reglas y la visualización del autómata en SVG.
  \item \texttt{test-manager/}: contiene \texttt{test-manager.tsx} y \texttt{use-test-manager.ts}, responsables de cargar el catálogo de pruebas, ejecutar casos individuales o masivos y calcular métricas de sesión.
  \item \texttt{shell/}: agrupa los componentes estructurales que mantienen la navegación y el ensamblaje general de la interfaz.
\end{itemize}

El intercambio de datos y artefactos se concentra en archivos auxiliares compartidos:

\begin{itemize}[leftmargin=2em]
  \item \texttt{frontend/src/lib/api.ts}: implementa la capa de acceso HTTP y la lectura de respuestas JSON.
  \item \texttt{frontend/src/lib/download.ts}: gestiona la descarga de tokens, del lexer generado y de representaciones serializadas del autómata.
  \item \texttt{frontend/src/types/analysis.ts}, \texttt{frontend/src/types/generator.ts}, \texttt{frontend/src/types/test-manager.ts} y \texttt{frontend/src/types/shell.ts}: tipan las estructuras intercambiadas entre componentes, hooks y API.
\end{itemize}

Las consideraciones principales de diseño para esta capa fueron:

\begin{itemize}[leftmargin=2em]
  \item \textbf{Consumo por rutas relativas}: toda comunicación con el backend se realiza mediante \texttt{/api/*}, y la resolución final se concentra en \texttt{next.config.ts}.
  \item \textbf{Renderizado integrado}: la aplicación principal se monta desde \texttt{page.tsx} y el shell administra el cambio de módulo sin abandonar la interfaz.
  \item \textbf{Visualización especializada}: el analizador muestra tablas y errores estructurados, el generador añade matriz de transiciones y renderizado SVG, y el gestor de pruebas incorpora ejecución y seguimiento de estados.
  \item \textbf{Manejo local de descargas}: la exportación de resultados se resuelve del lado del cliente para no introducir rutas auxiliares adicionales en la API.
  \item \textbf{Soporte de pruebas}: archivos como \texttt{vitest.config.ts}, \texttt{vitest.setup.ts} y las pruebas ubicadas junto a cada feature permiten validar hooks, componentes y flujos de interacción del frontend.
\end{itemize}

En conjunto, el frontend queda definido como una capa formada por una entrada de aplicación, un shell de navegación, features independientes por módulo, utilidades compartidas para API y descargas, y tipos explícitos que documentan la integración con el backend.

\section{Integración entre Frontend y Backend}

La comunicación entre capas se realiza mediante una API HTTP basada en FastAPI. El frontend consume rutas relativas y opera sobre contratos JSON tipados, lo que garantiza independencia entre la presentación y la lógica de compilación.

\section{Módulos Funcionales}

\subsection{Dashboard}

El dashboard resume información general del sistema: cantidad de especificaciones disponibles, cantidad de casos de prueba, distribución por complejidad y estado de los módulos operativos. Su función es ofrecer una vista consolidada del ecosistema completo del proyecto.

\subsection{Lexical Analysis}

Este módulo permite seleccionar una complejidad, cargar una entrada desde archivo o editarla manualmente y ejecutar el análisis léxico con el backend real. El resultado se presenta como una tabla de tokens, un panel de errores léxicos estructurados y estadísticas básicas de ejecución.

\subsection{Lexical Generator}

El generador permite editar o cargar una especificación YALex, compilarla y explorar sus resultados. Para cada compilación, la interfaz presenta:

\begin{itemize}[leftmargin=2em]
  \item una regla activa seleccionable;
  \item la visualización del autómata correspondiente;
  \item la matriz de transiciones;
  \item la lista de tokens reconocidos;
  \item el código del lexer generado listo para descarga.
\end{itemize}

\subsection{Test Case Manager}

El administrador de casos de prueba consume un catálogo declarativo de entradas y especificaciones. Permite ejecutar casos individuales o lotes completos, registrar resultados de sesión y mostrar indicadores como porcentaje de éxito, ejecuciones acumuladas y casos pendientes de validación.

\section{Generación de Lexer Independiente}

Uno de los principios centrales del sistema es que el archivo Python generado sea completamente autosuficiente. El lexer exportado no depende del generador ni del paquete interno que lo produce. En su lugar, incorpora:

\begin{itemize}[leftmargin=2em]
  \item la representación serializada de los autómatas;
  \item las estructuras básicas de tokens y errores;
  \item la función de escaneo;
  \item la lógica de prioridad y \emph{longest match};
  \item el entrypoint configurado por la especificación.
\end{itemize}

Esta propiedad garantiza portabilidad y hace posible utilizar el lexer generado como artefacto independiente dentro de otras herramientas o proyectos.

\section{Casos de Prueba}

La plataforma incluye casos de prueba organizados por complejidad funcional.

\subsection{Complejidad Baja}

La especificación de baja complejidad reconoce identificadores, enteros, operadores aritméticos y asignación. Este nivel valida el comportamiento básico del scanner y la omisión correcta de espacios en blanco.

\begin{lstlisting}[style=yalexstyle, caption={Ejemplo representativo de entrada para complejidad baja.}]
variable + 5 * -11 = otraVariable
\end{lstlisting}

\subsection{Complejidad Media}

La complejidad media incorpora palabras reservadas, operadores de comparación y literales de cadena, ampliando el conjunto de patrones y el nivel de ambigüedad potencial en el proceso de matching.

\begin{lstlisting}[style=yalexstyle, caption={Ejemplo representativo de entrada para complejidad media.}]
if x1 >= 10 else while nombre == "abc"
\end{lstlisting}

\subsection{Complejidad Alta}

La complejidad alta modela un subconjunto más expresivo de un lenguaje imperativo, con tipos, delimitadores, comparaciones compuestas, operadores lógicos y literales de punto flotante.

\begin{lstlisting}[style=yalexstyle, caption={Ejemplo representativo de entrada para complejidad alta.}]
int x = 10;
float y = 20.5;
if (x <= y && x != 0) { return x; }
\end{lstlisting}

Cada nivel incluye entradas aceptadas y entradas con error, lo que permite validar tanto la identificación correcta de tokens como la detección precisa de fallos léxicos.

\section{Estrategia de Pruebas y Validación}

La calidad del sistema se valida mediante una estrategia de pruebas multinivel:

\begin{itemize}[leftmargin=2em]
  \item \textbf{Pruebas unitarias del backend}: verifican tokenización, parsing, cálculo de funciones de posición, construcción de AFD, minimización, normalización YALex y serialización.
  \item \textbf{Pruebas de integración del backend}: validan el flujo completo entre parser, compilador, runtime y generación de código.
  \item \textbf{Pruebas de contrato}: verifican la forma exacta de las respuestas de la API y su consistencia para el frontend.
  \item \textbf{Pruebas de componentes e integración del frontend}: validan shell, análisis léxico, generación, ejecución de casos y descargas.
  \item \textbf{Pruebas end-to-end}: validan la experiencia completa de usuario desde la interfaz web hasta la ejecución real del backend.
  \item \textbf{Validación manual}: comprueba paridad visual, carga de archivos, descargas y ejecución de escenarios representativos.
\end{itemize}

Esta estrategia permite evaluar la corrección del sistema tanto desde la perspectiva algorítmica como desde la experiencia de uso final.

\section{Resultados del Sistema}

La plataforma ofrece un resultado integral sobre cuatro ejes principales:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Compilación}: una especificación YALex se traduce a un conjunto de reglas compiladas con autómatas minimizados.
  \item \textbf{Ejecución}: una entrada textual se procesa con política de \emph{longest match}, devolviendo tokens, offsets, línea, columna y errores léxicos estructurados.
  \item \textbf{Visualización}: el sistema muestra autómatas por regla, matrices de transición y tokens reconocidos desde una interfaz interactiva.
  \item \textbf{Validación}: el conjunto de casos de prueba permite verificar el comportamiento del sistema para complejidades baja, media y alta.
\end{enumerate}

La combinación de estos resultados convierte al proyecto en una plataforma didáctica y técnica capaz de mostrar tanto la teoría subyacente como su ejecución práctica.

\section{Conclusiones}

\begin{enumerate}[leftmargin=2em]
  \item La plataforma demuestra que es posible integrar teoría de autómatas, compilación léxica y visualización interactiva en un solo sistema coherente.
  \item La construcción directa de AFD a partir del árbol sintáctico ofrece una base formal sólida y evita dependencias en etapas intermedias innecesarias.
  \item La minimización de estados mejora tanto la representación del autómata como la eficiencia del lexer generado.
  \item La separación entre backend Python, API FastAPI y frontend web permite combinar exactitud algorítmica con una experiencia de usuario moderna.
  \item La generación de un lexer autosuficiente refuerza el objetivo central del proyecto: producir analizadores léxicos reutilizables e independientes del entorno de compilación.
  \item La estrategia de pruebas multinivel aporta una validación rigurosa del comportamiento del sistema desde la lógica interna hasta la interacción completa de usuario.
\end{enumerate}
